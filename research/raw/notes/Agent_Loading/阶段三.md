## 结构化输出链 

```python
from langchain_core.prompts import ChatPromptTemplate

from pydantic import BaseModel

from langchain_core.output_parsers import PydanticOutputParser

from langchain.chat_models import init_chat_model

class Entities(BaseModel):

    user_role: str | None = None

    goal: str | None = None

    current_stage: str | None = None

  

class IntentEntity(BaseModel):

    intent: str

    entities: Entities

  
  

parser = PydanticOutputParser(pydantic_object=IntentEntity)

  
  

prompt = ChatPromptTemplate.from_template("""

你必须严格按照JSON格式输出，字段名必须完全一致:

- intent

- entities                                                                                  

始终提取用户问题中的:

- 意图

- 实体:

    - user_role

    - goal

    - current_stage                                                                                                                              

  

用户输入: {input}

  

{format_instructions}                                                                                                                              

""").partial(format_instructions = parser.get_format_instructions())

  

model = init_chat_model("qwen2.5:7b",model_provider="ollama")

  

#LCEL

chain = prompt | model | parser

  

result = chain.invoke({"input": "我是一个大学生，现在想要学习agent开发，目前学习到LangChain基础，什么时候可以学完？"})

print(result)
```

### 项目分析：结构化输出链

#### 结构
- **核心组件**：Pydantic模型定义 + LangChain提示模板 + LCEL链式调用
- **数据模型**：
  - `Entities`：用户角色、目标、当前阶段
  - `IntentEntity`：意图 + 实体组合
- **技术栈**：LangChain Core + Ollama本地模型（qwen2.5:7b）

#### 功能
1. 从用户自然语言输入中提取结构化信息
2. 使用Pydantic进行输出验证和解析
3. 通过LCEL（LangChain Expression Language）实现链式调用
4. 支持本地模型推理，无需云端API

#### 待优化部分
- **错误处理缺失**：没有捕获模型输出格式错误或解析失败的异常
- **提示词硬编码**：JSON格式要求写在提示词中，应该完全依赖`format_instructions`
- **缺少输入验证**：未对用户输入进行长度或内容检查
- **可扩展性不足**：实体字段固定，难以适配其他场景
- **建议**：添加fallback机制、日志记录、支持自定义实体schema



## RAG

检索增强生成

索引阶段：
- 输入文档，PDF等数据
- 清理文档，对文档去噪，去重无用内容
- 增强文本，利用附加数据和元数据为文档分片提供上下文信息
- 文档拆分，使用文本分割器将文本分成较小的文档片段，严格适配嵌入模型和生成模型的上下文窗口限制
- 向量化表示，使用嵌入模型的嵌入函数，将文本片段映射为向量表示（也就是高维稠密向量）
- 存储到向量数据库，将生成后的向量和原始数据存储到数据库中

![索引阶段](v-0.svg)

检索阶段：
- 接收请求，接收用户的自然语言查询，先对其进行改写或扩充，提高后续检索的覆盖度
- 查询向量化，使用向量函数将查询转化为语义向量表示
- 信息检索，在嵌入存储中，通过语言相似度搜索找到与查询向量最相关的文档片段（通常是看余弦相似度和欧几里得相似度）
- 生成增强，将检索到的相关文档片段和原始查询作为上下文传给LLM，使用合适的提示词引导LLM基于检索的信息回答问题
- 输出结果，向用户输出自然语言回复，附带文档参考链接

### 项目分析：本地PDF问答系统

#### 结构
- **文档处理流程**：PyPDFLoader → RecursiveCharacterTextSplitter → 向量化 → InMemoryVectorStore
- **检索链路**：用户查询 → 相似度检索 → 文档链 → 检索链 → 生成答案
- **技术栈**：LangChain Classic + Google Gemini Embeddings + Ollama LLM

#### 功能
1. 加载PDF文档并按页提取文本
2. 使用递归字符分割器进行文本分块（chunk_size=200, overlap=50）
3. 通过Google Gemini嵌入模型生成向量
4. 基于相似度检索Top-3相关片段
5. 使用本地Ollama模型生成答案

#### 待优化部分
- **内存存储不持久**：使用`InMemoryVectorStore`，程序重启后数据丢失
- **分块策略简单**：chunk_size=200过小，可能导致上下文碎片化
- **依赖外部API**：Google Embeddings需要网络，与"本地"定位矛盾
- **缺少元数据**：未保存页码、章节等信息，答案无法溯源
- **无会话记忆**：每次查询独立，无法处理多轮对话
- **建议**：改用Chroma持久化、增大chunk_size至500-800、使用本地嵌入模型、添加元数据引用

def clean_rag_text(text: str) -> str:

    """

    专为 RAG 设计的文本清洗：

    - 修复因 PDF 排版导致的单词内换行（英文）

    - 合并中文段落内的软换行

    - 保留段落之间的空行作为自然分割标记

    """

  

    text = re.sub(r'(\w+)-\n(\w+)', r'\1\2', text)

  

     # 2. 对于中文：将单行换行符替换为空格（若下一行以中文或小写字母开头，视为同一段落）

    lines = text.split('\n')

    merged_lines = []

    for line in lines:

        line = line.strip()

        if not line:

            merged_lines.append('')  # 保留空行作为段落边界

            continue

        if merged_lines and merged_lines[-1] != '':

            # 判断是否应合并到上一行（中文或非首字母大写的英文）

            prev_line = merged_lines[-1]

            if (re.match(r'^[\u4e00-\u9fff]', line) or

                (re.match(r'^[a-z]', line) and not re.search(r'[.!?]$', prev_line))):

                merged_lines[-1] += ' ' + line

                continue

        merged_lines.append(line)

    return '\n'.join(merged_lines)

class MemoryManager:

    def __init__(self):

        self.sessions = {}  # {session_id: session_data}

  

    def create_session(self, user_id=None):

        """创建新会话，返回 session_id"""

        sid = str(uuid.uuid4())

        self.sessions[sid] = {

            "session_id": sid,

            "messages": [],

            "context_chunks": [],

            "metadata": {

                "created_at": time.time(),

                "last_updated": time.time(),

                "user_id": user_id

            }

        }

        return sid

  

    def add_message(self, session_id, role, content, chunks=None):

        """向会话添加一条消息"""

        if session_id not in self.sessions:

            return

        session = self.sessions[session_id]

        session["messages"].append({"role": role, "content": content})

        if chunks:

            session["context_chunks"].extend(chunks)

        session["metadata"]["last_updated"] = time.time()

  

    def get_history_text(self, session_id, max_turns=None):

        """获取会话历史文本（用于提示词）"""

        session = self.sessions.get(session_id)

        if not session:

            return ""

        msgs = session["messages"]

        if max_turns:

            msgs = msgs[-max_turns*2:]  # 一问一答算两轮

        lines = []

        for m in msgs:

            prefix = "用户" if m["role"] == "user" else "助手"

            lines.append(f"{prefix}: {m['content']}")

        return "\n".join(lines)

  

    def get_session(self, session_id):

        return self.sessions.get(session_id)

  

    def clear_session(self, session_id):

        if session_id in self.sessions:

            del self.sessions[session_id]

### 项目分析：PDF文档问答助手（生产级）

#### 结构
- **文档处理模块**：
  - `extract_text_for_rag`：页面级提取 + 元数据增强
  - `clean_rag_text`：中英文混合文本清洗
- **向量化模块**：SentenceTransformer + 批量处理 + 重试机制
- **存储层**：Chroma向量数据库（HTTP客户端）
- **记忆管理**：`MemoryManager`类实现会话级记忆
- **问答引擎**：`ask_question`函数整合检索+历史+LLM

#### 功能
1. **智能文本清洗**：修复PDF排版问题（连字符换行、段落合并）
2. **语义分块**：800字符块 + 200重叠 + 多级分隔符（段落→句子→标点）
3. **批量向量化**：32条/批 + 指数退避重试 + 归一化处理
4. **持久化存储**：Chroma集合复用，避免重复向量化
5. **会话记忆**：支持多轮对话，保留最近3轮历史
6. **元数据溯源**：答案附带页码信息
7. **交互式CLI**：循环问答，支持退出命令

#### 待优化部分
- **硬编码配置**：文件路径、模型名称、API密钥散落各处，应使用配置文件
- **单线程处理**：向量化和检索未使用异步，大文档处理慢
- **记忆策略简单**：仅保留最近3轮，缺少摘要压缩或向量记忆
- **无PII保护**：未实现敏感信息脱敏（阶段三要求的中间件功能）
- **缺少评估指标**：无检索准确率、答案质量评分
- **错误处理不完善**：LLM调用失败后仅返回错误信息，未尝试降级策略
- **无流式输出**：长答案需等待完整生成
- **改进建议**：
  - 使用`pydantic-settings`管理配置
  - 引入`asyncio`实现并发向量化
  - 添加LangChain 1.0中间件（PII隐蔽、摘要化）
  - 实现混合检索（向量+关键词）
  - 添加LangSmith追踪
  - 支持流式响应（SSE）

## 三个项目对比总结

| 维度          | 项目一   | 项目二 | 项目三       |
| ----------- | ----- | --- | --------- |
| **复杂度**     | 简单    | 中等  | 高         |
| **生产就绪度**   | Demo级 | 原型级 | 接近生产      |
| **持久化**     | 无     | 无   | 有（Chroma） |
| **记忆能力**    | 无     | 无   | 有（会话级）    |
| **错误处理**    | 缺失    | 基础  | 较完善       |
| **可扩展性**    | 低     | 中   | 高         |
| **符合阶段三要求** | 部分    | 部分  | 大部分       |

**总体评价**：项目三最接近阶段三目标（LangChain 1.0 + RAG + 记忆），但仍需补充中间件系统、异步处理和可观测性才能达到"精通"水平。
