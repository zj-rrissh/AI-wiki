## 方法名称

评测构建路线图

## 解决的问题

如何从零开始构建可靠的 Agent 评测系统。避免常见陷阱（过度设计、评分偏见、任务歧义）。

## 四步构建法

### Step 0：起步

> **20-50 个简单任务即可起步，不要等"几百个任务"**

- 从简单任务开始
- 快速迭代，不要追求完美
- 验证评测基础设施是否工作

### Step 1：转化失败

> **将手动检查/用户报告失败转化为测试用例**

- 生产环境中发现的问题 → 测试用例
- 用户报告的 bug → 回归测试
- 手动 code review 发现的模式 → 自动检查

### Step 2：消除歧义

> **任务要无歧义，两位领域专家独立判断应得相同结论**

- 编写任务描述时假设读者不了解上下文
- 让两位专家独立评审
- 不一致 → 重写任务描述

### Step 3：平衡问题集

> **既测"应该发生"也测"不应该发生"**

- 正向测试：验证正确行为
- 负向测试：验证拒绝错误输入
- 边界测试：极限情况处理

### Step 4：隔离 trial

> **每次从干净环境开始，避免状态污染**

- 每次尝试独立的容器/环境
- 清理临时文件、数据库状态
- 独立的端口、配置文件

## 关键术语

| 术语 | 定义 |
|------|------|
| **Task/Problem** | 单次测试，有定义输入和成功标准 |
| **Trial** | 一次尝试 |
| **Grader** | 评分逻辑，每个 task 可有多个 grader |
| **Transcript/Trace** | 完整记录（API 调用+返回+推理） |
| **Harness** | 运行基础设施 |

## Grader 类型选择

| 类型 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **代码级** | 快速、客观、可复现 | 需要精确的预期输出 | 确定性任务 |
| **模型级** | 灵活、可扩展、处理开放式输出 | 可能有评分偏见 | 开放式任务 |
| **人类级** | 黄金标准、校准模型 grader | 慢、成本高 | 最终验证 |

## pass@k vs pass^k

- **pass@k**：k 次尝试中至少一次成功。k↑ 时 pass@k↑
  - 适用：工具类任务（关心"至少一次成功"）
- **pass^k**：k 次尝试全部成功。k↑ 时 pass^k↓
  - 适用：用户Facing 任务（关心"每次都可靠"）

## 常见陷阱

1. **Eval 过度设计**：团队可能在构建 eval 上花费数周，比实际构建功能还多
2. **资源配置混淆**：终端用户在 leaderboard 上看到的分数差异可能反映硬件差异而非模型能力
3. **评分偏见**：即使"客观"的代码测试也可能因 grading bug 产生 40%+ 的分数差异
