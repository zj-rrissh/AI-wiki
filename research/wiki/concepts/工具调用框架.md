# 工具调用框架

一种让 LLM 主动选择并执行外部工具的架构模式。

## 定义

通过 JSON Schema 定义工具接口，让模型自主判断是否调用工具、调用哪个工具、传入什么参数。

## 核心组件

```
用户输入 → Prompt 构建 → LLM 推理 → JSON 解析 → 工具执行 → 结果返回 → LLM 总结
```

### 工具定义示例

```json
{
  "name": "get_weather",
  "description": "获取指定城市的天气信息",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "要查询天气的城市名称"
      }
    },
    "required": ["city"]
  }
}
```

### 决策流程

1. **Prompt 构建**：将工具列表注入 system prompt
2. **LLM 推理**：模型输出 JSON（tool + args）或纯文本（answer）
3. **JSON 解析**：`json.loads()` 或正则提取
4. **工具执行**：通过 tool_map 路由到具体函数
5. **结果注入**：将工具返回传给 LLM 生成最终回答

## 设计模式

| 模式 | 特点 | 适用场景 |
|------|------|----------|
| JSON Schema 驱动 | 手动定义工具接口 | 固定工具集 |
| OpenAI Function Calling | API 原生支持 | 云端模型 |
| LangChain @tool 装饰器 | 自动元数据 + 运行时注入 | LangChain 生态 |

## 关键数据点

- **工具数量**：建议 3-5 个，避免决策疲劳
- **超时处理**：Ollama 本地模型可能响应慢
- **降级策略**：工具执行失败时返回错误信息，不阻塞对话

## 前提与局限性

- **模型能力依赖**：小模型（如 qwen2.5:7b）工具调用准确性有限
- **JSON 解析脆弱**：需要正则 fallback 处理格式错误
- **无状态**：每次调用独立，不保留上下文
