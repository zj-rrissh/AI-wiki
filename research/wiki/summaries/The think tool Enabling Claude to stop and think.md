## 核心结论

1. **"思考"工具让 Claude 在执行工具调用链过程中增加额外的分析步骤**，专注于模型新发现的信息，与扩展思维（生成响应前深思考）在时序和目的上不同。

2. **τ-Bench 航空领域：优化提示的"思考"工具达到 0.570 pass^k=1，基线仅 0.370**（相对提升 54%）。零售领域：单独"思考"工具达到 0.812，基线 0.783。

3. **"思考"工具最佳场景**：工具输出分析、策略繁多环境、顺序决策（每步建立在前一步上且错误代价高昂）。不适用于非顺序工具调用或简单指令执行。

## 支撑证据

- τ-Bench 使用 pass^k 指标（所有 k 次试验均成功概率），不同于 pass@k，评估一致性和可靠性
- SWE-bench 上添加"思考"工具，n=30 vs n=144 对照，平均性能提升 1.6%（Welch t 检验 p<.001, d=1.47）
- 航空领域从"思考"示例获益最大，因为该领域政策高度复杂

## 关键方法

### 思考工具定义
```json
{
  "name": "think",
  "description": "Use to think about something. Will not obtain new information or change the database, but just append thought to log.",
  "input_schema": {
    "type": "object",
    "properties": {
      "thought": {"type": "string", "description": "A thought to think about."}
    },
    "required": ["thought"]
  }
}
```

### 最佳实践
1. **提供领域特定示例的策略性提示**：在系统提示中给出何时/如何使用"思考"工具的具体示例
2. **在系统提示而非工具描述中添加复杂指导**：为"思考"工具的说明提供更广泛上下文
3. **观察并迭代**：监控 Claude 实际如何使用，调整提示鼓励更有效思维模式

### 性能数据（τ-Bench 航空域）

| 思考+提示 | 0.584 | 0.444 | 0.384 | 0.356 | 0.340 |

## 质疑

- **对简单领域提升有限**：零售领域政策更简单，仅提供思考空间就足够，无需示例提示
- **增加 token 开销**：每次思考都会消耗额外 token，在高频率调用场景可能不实际
- **不适用于所有任务类型**：非顺序或简单任务可能因额外思考而变慢且无收益

## 跨领域迁移

- **复杂客服/合规场景**：需要遵循详细策略且决策有链式依赖的场景，"思考"工具价值最大
- **代码调试**：在发现 bug 源后，调用思考工具 brainstorming 多种修复方案并评估
- **多步骤规划**：每步决策建立在前序结果上时（如金融建模、科学研究），思考工具可防止错误级联
