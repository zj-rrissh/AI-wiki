## 核心结论

1. **上下文工程是提示工程的自然延伸**，核心问题从"如何写好提示"扩展为"如何为 LLM 管理最优 token 集合以持续达成预期结果"。

2. **注意力稀缺性是 LLM 的架构约束**：n 个 token间存在 n² 成对关系，上下文中 token 越多，模型关注特定信息的能力越低（"上下文衰减"现象）。

3. **三类长期任务上下文管理技术**：压缩（compaction）、结构化笔记（Agent Memory）、多 Agent 架构，各有适用场景。

## 支撑证据

- "大海捞针"基准研究揭示上下文衰减：token 数增加时，模型准确回忆上下文信息的能力下降
- 位置编码插值技术使模型能处理更长序列，但会损害对 token 位置的理解
- 克劳德玩宝可梦案例：Agent 精确记录数千步游戏进程，跨会话保持目标连贯性

## 关键方法

### 系统提示校准的"黄金地带"
- **过度具体（脆弱）**：硬编码 if-else 逻辑精确引导行为 → 系统脆弱+维护复杂
- **过于笼统（无效）**：模糊指导无法提供期望输出的具体信号
- **最佳点**：足够具体以引导行为，足够灵活以提供鲁棒启发式

### 动态上下文检索策略
- **预检索**：静态 embedding-based 预推理时检索
- **即时检索（JIT）**：Agent 维护轻量标识符（文件路径/链接），运行时按需加载
- Claude Code 混合模型：CLAUDE.md 预加载 + glob/grep 导航环境

### 三大长期任务技术对比
| 技术 | 适用场景 | 核心机制 |
|------|----------|----------|
| 压缩 | 需要大量对话来回的任务 | 消息历史→摘要→新 context window |
| 结构化笔记 | 有清晰里程碑的迭代开发 | Agent 定期写笔记到外部存储 |
| 多 Agent | 复杂研究和分析（可并行探索） | sub-agent 返回精简摘要（1000-2000 token） |

## 质疑

- **压缩的信息选择难题**：过度压缩会丢失微妙但重要的上下文，且这类重要性往往在后续才显现
- **运行时探索的效率 vs 精确权衡**：JIT 探索比预计算数据慢，且需要精心设计工具和启发式引导 Agent
- **混合策略的边界模糊**：最佳自主程度因任务而异，没有明确界定

## 跨领域迁移

- **渐进式信息披露**适用于人类认知科学中的"工作记忆有限"理论
- **子 Agent 架构**与操作系统虚拟化思想同源：抽象接口使实现可替换
- **元认知工具（think tool）**的思路可迁移到其他需要阶段性自我反思的 Agent 场景
