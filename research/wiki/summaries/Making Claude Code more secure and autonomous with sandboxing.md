## 核心结论

1. **沙箱通过文件系统隔离 + 网络隔离双重边界实现安全提升**：即使提示注入成功，攻击者也被完全隔离在预设边界内，无法窃取 SSH 密钥或连接攻击者服务器。

2. **沙箱减少 84% 的权限提示**：通过定义 Claude 可自由工作的边界，使安全操作自动允许，恶意操作自动阻止，仅在越界时才请求权限。

3. **文件系统 + 网络隔离必须同时实现**：缺任一者都会被对方弥补——无网络隔离则被入侵的 Agent 可通过外部传输文件；无文件系统隔离则 Agent 可逃逸获得网络访问。

## 支撑证据

- Linux 用 bubblewrap，macOS 用 seatbelt，在 OS 层面强制执行限制
- 沙箱不仅覆盖 Claude Code 直接交互，还包括命令生成的脚本/程序/子进程
- Web 版 Claude Code 使用自定义代理服务处理所有 Git 交互，签名密钥永不进入沙箱

## 关键方法

### 沙箱架构两组件
1. **文件系统隔离**：允许对当前工作目录读写，阻止修改目录外任何文件
2. **网络隔离**：只允许 Unix 域套接字连接到沙箱外代理服务器，代理限制可连接的域并处理新域的用户确认

### 安全属性
- **机密性保护**：SSH 密钥等敏感凭证不会被窃取
- **完整性保护**：防止下载恶意软件或非预期代码执行
- **隔离保证**：即使 Claude Code 被攻破，攻击影响也限制在沙箱内

### Web 版安全设计
- 每个会话运行在独立沙箱
- Git 客户端使用自定义作用域凭证向代理服务认证
- 代理验证凭证 + Git 交互内容，然后才附加正确认证令牌

## 质疑

- **沙箱配置复杂度**：开发者需要理解边界定义才能有效使用，对于新手可能过于复杂
- **性能和安全的权衡**：沙箱引入的 OS 层面限制可能有性能开销
- **代理服务的信任假设**：用户需要信任沙箱外部的代理服务不会成为攻击向量

## 跨领域迁移

- **云原生安全模型**：借鉴容器化安全思路，将"pets vs cattle"理念应用于 Agent
- **代理模式**：可用于其他需要安全网络控制的应用场景
- **最小权限原则**：Agent 的权限应限制在完成任务所需的最小范围
